3.1 Architecture: Hub vs. Project

VenvHub Pro uses an architecture that strictly separates your source code from virtual environments. It is important to understand how this works in the background:

  • Projects: The application only "reads" folder names in your Projects Root. It never writes anything to these folders (except for requirements.txt upon request or .vscode/settings.json for integration).
  • Venv Hub: All virtual environments are physically stored in a central folder called Venv Hub.
Advantage: If you accidentally delete the entire virtual environment, your source code (project) remains completely untouched. These two things are physically separated on the disk.

3.2 Creation and Naming

When creating a new environment, the application uses a strict naming convention and intelligently selects the technology based on the Python source.

Creation and Naming Process:

  1. You select a project (e.g., MyEshop) and a Python version (e.g., [Local] 3.12).
  2. You enter an environment name, e.g., My new version!.
  3. Your entered name passes through an internal sanitize_venv_name filter. Result: My_new_version!.
  4. The application creates a folder in the Hub directory named:
    MyEshop_My_new_version!.
What does sanitize_venv_name do?
This filter prepares a safe name for a Windows folder. It replaces spaces with underscores and removes prohibited characters (< > : " / \ | ? *), but intentionally leaves diacritics, exclamation marks, and characters from other languages (e.g., Cyrillic) to ensure global usability.

Backend Technologies (System vs. Local)

Depending on the selected Python type, a different technology is used:

  • System Python: Uses the built-in venv module. It is a standard and reliable method that creates a lightweight environment. However, this environment is hard-linked to the main Python installation on your PC, making it non-portable.

  • Local Python (Embeddable): Uses the more advanced virtualenv library. It is specifically designed to work with portable versions and can create a fully self-sufficient and isolated environment. This is the ideal choice for portability.
Threading: The entire creation process runs in a separate thread (QThread). This means that even if environment creation takes 10 seconds, the application won't "freeze," and you can continue working.

3.3 Row Actions and Process Management

Each row in the table represents one physical environment. The buttons in the "Actions" column have direct access to system processes.

▶ Play (Secure Launch via temporary .BAT)

When you click Play in a row, the application does not rely on long and fragile cmd.exe commands chained with &&, which would fail with spaces in folder names. Instead, it generates a temporary batch file (.bat) in the background:

  1. A secure file is created in the Temp folder (e.g., venvhub_run_123.bat).
  2. Instructions are written on separate lines: directory change to the project, venv activation, and execution of your script.
  3. This file is launched via cmd.exe /k. The window remains open after the script ends so you can see any errors.

■ Stop (Process Registry & Watchdog)

VenvHub Pro includes an internal Process Registry system. When you launch a script, the application remembers its PID (Process ID) and pairs it with the venv path. Simultaneously, a "Watchdog" connects to this process.

When you press Stop:

  1. The application looks in the registry to see which PIDs are running under this venv.
  2. It finds not only the main process (cmd.exe) but also all its "children" (python.exe, subprocesses).
  3. It performs a Kill Tree - forcibly terminating the entire process tree so nothing remains hanging in memory (via the psutil module).
What does the Watchdog do?
This background process constantly monitors the running script. If the script crashes on its own (e.g., due to a code error), the Watchdog detects it and reports an "Error!" status to the main application, even if you haven't pressed the Stop button.

3.4 Logical Linking and Scripts

Default Venv (star ★)

A project can have multiple environments, but only one is "primary." Clicking "Default" writes this information to the venv_hub_pro_config.json file. This setting is permanent and is used by tools like the Mini-Bar or VS Code integration.

Entry Script Selection (Validation)

The application includes a safety check when choosing a .py file. It uses the os.path.relpath function to verify the path.

Rule: You can only select a script that is physically located inside the project folder. If you try to select a file from another drive, the application will reject it to maintain project portability on USB drives.

3.5 Cloning (Intelligent Reconstruction)

The "Clone" function (right-click on a row) is not a simple folder copy. It performs an intelligent reconstruction in the background, compatible even with the fast UV installer:

  1. Freeze: Retrieves the package list from the old environment.
  2. Create: Creates a brand new, clean environment.
  3. Install: Installs all packages from the list into the new environment in isolated mode (--no-build-isolation for stability).

Cloning Strategy by Venv Type

The application automatically chooses the best strategy during cloning:

Scenario 1: Cloning a System Venv

The application uses the same system Python that created the original (e.g., C:\Python311\python.exe) to create the new environment. The clone will also be system-based and non-portable.

Scenario 2: Cloning a Local/Embed Venv (Plan A - Fast)

If the original "parent" folder with the portable Python (Local) still exists, the application reads the path from the "Birth Certificate" and uses this parent Python to directly and very quickly create the clone via virtualenv. This is the fastest and most reliable method.

Scenario 3: Cloning a Local/Embed Venv (Plan B - Backup)

If you have deleted the parent Python folder in the meantime, the application has a backup plan:

  1. It temporarily installs the virtualenv library directly into the environment you want to clone.
  2. It uses the python.exe from inside that environment to run virtualenv and create the new clone.
This method ensures that cloning always succeeds, even if the original source no longer exists.

3.6 Venv Validator (Diagnostics and Repair)

VenvHub Pro includes an intelligent background system that protects your environments from errors after movement or disk changes.

Why are system venvs unstable?
The answer lies in the pyvenv.cfg file inside the environment. For a system venv, it contains a fixed, absolute path to the main Python installation, e.g., home = C:\Python311. If you move this environment to another PC, the link becomes invalid.

How does the Validator work?

Every time the table is displayed, the application compares the path in pyvenv.cfg with reality. Based on the diagnostics, it assigns a status:

  • VALID: Everything is in order.
  • BROKEN - FIXABLE: The path is wrong, but a compatible Python version exists on the PC (it can handle downgrades or upgrades of the third patch number). An automatic environment repair is offered without deleting packages.
  • BROKEN - INCOMPATIBLE: The required Python version (same major/minor branch) is missing from the PC. Reconstruction using a different version is offered.

3.7 Environment Birth Certificate (venv_birth_certificate.json)

Every environment created via VenvHub Pro contains a venv_birth_certificate.json file. This file is the brain behind the application's advanced features.

What information does it contain?

  • Project Name: Which project the environment logically belongs to.
  • Python Version: The exact version used during creation (e.g., 3.12.4).
  • Parent Python Path: Path to the python.exe from which the environment was created. This is crucial for fast Plan A cloning.
  • Package List: A complete list of installed packages in JSON format, which updates automatically with every change in the Pip Manager.
  • Creation and Modification Dates: For better organization.
Important: Thanks to this file, VenvHub Pro does not need to run slow commands (like pip freeze) during operations like Cloning or EXE generation. It can load all information instantly. Never delete this file.

3.8 VS Code Integration (Automatic Sync)

One of the most common developer problems is creating a virtual environment but having the code editor (Visual Studio Code) unaware of it, highlighting imports in red, and failing to provide code completion (Linter/IntelliSense not working).

VenvHub Pro solves this problem completely automatically through silent integration. It communicates directly with the hidden .vscode/settings.json folder inside your project.

1. Creating a New Environment (Safe Write)

When you click + CREATE in VenvHub Pro, the application checks your project after successful creation. If it finds that VS Code hasn't been assigned an environment yet, it automatically assigns it. If you already have an environment set manually in VS Code, VenvHub Pro will not overwrite it out of respect.

2. Changing the Default Environment (★ Button)

If you click the "Default" button (star) in the VenvHub Pro table, you tell the application: "This is now the primary environment for this project."

VenvHub Pro immediately sends this information to VS Code and forcibly overwrites the path in settings.json (key python.defaultInterpreterPath). Once you open VS Code, it will immediately start using this new environment for code completion.

Intelligent Local Package Sync: Another crucial thing happens in the background. When you change the default Venv, the application "scans" it. If it finds that this new Venv contains linked Local Packages, it instantly pushes them to VS Code as well (python.analysis.extraPaths). If it doesn't, it clears VS Code of any old local packages from the previous Venv to avoid chaos and "fake" suggestions in your editor.

3. Deletion and Cleanup

If you ❌ Delete an environment through VenvHub Pro that was also the default for VS Code, the application not only deletes files from the disk but also cleans up after itself in the editor. It opens settings.json, removes invalid paths, and if nothing else remains in the file, it deletes the entire file and the .vscode folder to keep your project tidy.

Portable Mode and Path Repair:
The highlight of the integration comes in Portable mode. If you have a project and environment on a USB drive and move it to another computer, the drive letter changes (e.g., from E: to F:).

VS Code would normally get confused because it would still look for the environment on drive E:. In portable mode, VenvHub Pro silently opens VS Code settings after running the repair and overwrites the old drive letter with the new one (via the PortablePathLogic module). You just open the editor, and everything works perfectly!

4. Secure Merging and Error Handling

The application accesses the .vscode/settings.json file in a non-destructive manner. This means it never overwrites the entire file with its own version but performs intelligent merging at the key level.

  • Preservation of User Settings: When modifying settings, the existing file is first loaded and parsed. The application only modifies or adds the keys it manages. Other settings (such as formatting, tabs, or other extensions' configurations) remain untouched.
  • Managed Interpreter Keys:
    • The python.defaultInterpreterPath key is set to the selected venv, and python.terminal.activateEnvironment is set to true.
    • Simultaneously, redundant keys python.venvPath and python.envFile are removed to clean up old configurations.
  • Selective Path Analysis Management: When writing local packages to the python.analysis.extraPaths key, the application does not remove your own custom paths. It filters the list, removing only paths that belong under the managed venv_hub_root folder, and then appends the new links.
  • Handling Corrupted Files (Fail-safe):
    • If settings.json is syntactically corrupted (e.g., a missing brace) and reading fails (json.JSONDecodeError), the application doesn't fail; instead, it creates a new correct structure to write the required parameters.
    • Any disk restrictions or missing access rights are handled at the system exception level (OSError), ensuring the entire interface doesn't crash.

3.9 Advanced Tools: VS Code Meta-Sync

VenvHub Pro allows you to create custom "project modules" (Local Packages / Linker) that automatically inject not only Python code but also predefined Tasks and Keybindings into Visual Studio Code.

How does it work?

If you create a package in the Local Packages folder containing a local_meta.json file, VenvHub Pro checks its content every time it is linked to a Venv (Apply Changes). If it finds a vscode_integration section, it writes these settings directly into your project (tasks.json and keybindings.json files).

Advantage: You don't have to manually configure tasks.json in every new project. Just link your local tool to the environment, and VenvHub Pro will do it for you.

Example: AI Sync Tool

Imagine a package used for comparing code changes using AI. In local_meta.json, you define:

{
    "name": "AI VS Code Sync",
    "vscode_integration": {
        "tasks": [
            {
                "label": "AI: Open Diff Editor",
                "type": "shell",
                "command": "${command:python.interpreterPath}",
                "args": ["-m", "ai_sync", "prepare", "${file}"]
            }
        ],
        "keybindings": [
            {
                "key": "ctrl+k d",
                "command": "workbench.action.tasks.runTask",
                "args": "AI: Open Diff Editor"
            }
        ]
    }
}

Security and Intelligent Synchronization

  • No duplicate settings: The validator checks if a Task or Shortcut already exists in the project before writing. If so, it skips them to avoid breaking your custom modifications.
  • Uses correct Python: Tasks are automatically bound to ${command:python.interpreterPath}, ensuring they run inside the currently active virtual environment (which you set as default in VenvHub).
  • Detailed logging: You can monitor the entire synchronization process in the log window when saving changes (you see exactly which task was added and which was skipped).
Note for Developers: This feature turns your local packages into true editor plugins. If your Python module needs specific control within VS Code, local_meta.json is the way to deliver it to the end user without any manual setup.
