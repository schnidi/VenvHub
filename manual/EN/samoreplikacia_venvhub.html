14. Architecture, Compilation, and Self-Replication

VenvHub Pro is not just a standard Python application. Thanks to unique core modifications, it possesses an advanced architecture that allows it to run on different graphical libraries and even compile "itself" (self-replication).

14.1 Dual-Framework (Compatibility Bridge)

Python applications usually must be written for either PyQt6 or PySide6. VenvHub Pro solves this using a Compatibility Bridge (core/logic/pyqt_to_pyside.py). This bridge utilizes the MetaPathFinder technique.

  • Upon launching main.py, Venv environment detection is activated.
  • If the application finds PyQt6, it runs natively.
  • If it finds PySide6, the bridge is inserted into the system memory. As soon as Python encounters a from PyQt6... import, our interceptor steps in, translates the request, and returns PySide6.
Result: The source code remains 100% written for PyQt6, but the software automatically rewrites all imports, signals, and slots to PySide6 at runtime if necessary.

14.2 Magic of .UI Files

The difference between frameworks also lies in how XML designs (.ui) are loaded.

  • PyQt6 (uic.loadUi): Injects buttons directly into the existing window (self).
  • PySide6 (QUiLoader): Always creates a brand new window.

VenvHub Pro contains a so-called Mimic function. If running under PySide6, this function lets QUiLoader create a new window, rips all widgets and the Layout out of it, and "pastes" them onto your original self instance. It also correctly sets the working directory so relative paths to icons are not lost.

14.3 VenvHub Self-Replication

Because VenvHub Pro features an integrated PyInstaller Builder tool, you don't need any external scripts to compile VenvHub itself.

  1. Open the running VenvHub Pro.
  2. In the Settings tab, select the projects root where the VenvHub code is located.
  3. Go to the PyInstaller Builder tab and select main.py as the entry script.
  4. Set Assets and parameters (see the section below).
  5. Click BUILD .EXE.
This approach (dogfooding) guarantees that the builder in VenvHub is fully functional, as it can safely compile even a massive application like itself.

14.4 EXE Creation (PyInstaller)

Since PyInstaller works on the principle of static code analysis, compilation differs slightly depending on which graphical library you have in your .venv.

General rule for Assets (Tab 2):
You MUST always include the following folders (source = target): ui, core/themes, translations, assets, navod.

Branch A: Compilation with PyQt6

When analyzing the main.py file, PyInstaller directly sees the imports and automatically bundles all necessary DLL libraries for PyQt6 into the .exe. You don't need to set anything in the Imports (3) tab.

Branch B: Compilation with PySide6

From the code's perspective, PyQt6 is written everywhere. PyInstaller won't notice PySide6. We must explicitly command it in the 3. Imports tab:

  • Hidden Imports: Add sequentially: PySide6, PySide6.QtCore, PySide6.QtGui, PySide6.QtWidgets, PySide6.QtNetwork, PySide6.QtUiTools.
  • Exclude Modules: Add PyQt6 (to avoid bundling unnecessary baggage if traces of it remain).

After launching the resulting .exe, our "Bridge" activates and seamlessly connects the source code with the bundled PySide6.

14.5 Installer Creation (NSIS)

Regardless of the branch used, the compilation result will be the dist/VenvHubPro folder (when choosing --onedir). This folder contains everything necessary to run the application.

To create an installer using the NSIS system, just pack this entire folder into a script:

# Insert into your .nsi script:
File /r "path\to\project\dist\VenvHubPro\*"

NSIS doesn't care which framework is running inside. It simply compresses (LZMA compression) the entire folder into a single Setup.exe file and creates desktop shortcuts for the end user.
