10. Local Packages (Shared Libraries)

10.1 Operating Principle (No UAC)

The Local Packages feature solves a common developer problem: how to efficiently share your own code (libraries, modules) between multiple projects without the need for constant copying or installation.

VenvHub Pro requires no administrator rights (UAC) for linking and does not create physical copies or directory junctions on the disk. Instead, it utilizes an exceptionally clean, fast, and secure Python Import Hooks (sys.meta_path) architecture. The system automatically handles:

  • Creating the mapping file venvhub.json in the site-packages folder of the target Venv.
  • Inserting a dynamic loader (.pth and .py file) that intercepts your imports at Python startup and directs them straight to your code.
  • Automatic silent installation of required external (PIP/UV) dependencies for the package.
  • Complete injection of absolute paths and tasks into VS Code (IntelliSense).

10.2 Package Structure (local_meta.json)

For VenvHub Pro to work with your package, every folder in the archive must contain a local_meta.json configuration file. This file defines the name, version, and requirements of the package.

Metadata Entry Example:

{
    "name": "MyDatabase",
    "version": "1.2.0",
    "short_description": "Provides connection to MongoDB",
    "requires_local": ["MyLogger"],
    "requires_pip": ["colorama", "pymongo"],
    "vscode_integration": {
        "tasks": [
            {
                "label": "Clear Database Cache",
                "type": "shell",
                "command": "python -c \"import db; db.clear()\""
            }
        ],
        "keybindings": [
            {
                "key": "ctrl+shift+d",
                "command": "workbench.action.tasks.runTask",
                "args": "Clear Database Cache"
            }
        ]
    }
}
Key Meaning
short_description A short description displayed directly in the VenvHub Pro table.
requires_local List of names of other folders in the same archive that this package requires.
requires_pip List of PyPI packages to be automatically installed.
vscode_integration Allows defining custom commands (Tasks) and keybindings that are automatically added to the target project.

10.3 Root Path Setup

Before the first use, you must tell the application where your local code archive is located:

  1. Open Manager (🖥️) -> Settings tab.
  2. In the Basic Paths section, find the Local Packages entry.
  3. Click the folder icon 📁 and select the folder where you keep your modules (each module in its own subfolder).
Tip: If you are using the Portable version on a USB drive, the path automatically adjusts to the current drive letter.

10.4 Linking into Venv

The actual process of adding a package to a specific environment is as follows:

  1. In the environment table, right-click on the selected Venv.
  2. Select the 🔗 Local Packages option.
  3. In the window that opens, check the packages you want to use in the project.
  4. Click 💾 Save Changes (Relink).
Dependency Automation: If you check "Package A" which requires "Package B", the system automatically reads its `local_meta.json`, writes both paths into the `venvhub.json` in the Venv, and silently installs all necessary PIP packages.

10.5 Integration with VS Code

One of the primary advantages of VenvHub Pro is coordinated synchronization with the VS Code development environment. When you make a change to the local package settings or switch the active virtual environment, the system performs the following operations:

  • VenvHub Pro reads the venvhub.json file in the given environment and extracts the absolute paths of the linked local libraries.
  • It sends these paths to the python.analysis.extraPaths configuration array in your VS Code project's settings.json file. Consequently, Pylance and IntelliSense recognize your imports, preventing false error highlights.
  • Custom tasks (tasks.json) and keybindings (keybindings.json) are attached to the project based on the metadata configuration.
🛡️ Hybrid Mechanism (Brutal Check) and the venvhub_tracker.json File:

To prevent VenvHub Pro from unintentionally modifying tasks and keybindings that you created manually in VS Code, the system utilizes a combined verification mechanism during package removal:

  1. Write Registry (Receipt): During each integration of tasks or keybindings, the application records the exact list of performed modifications in a helper file named .vscode/venvhub_tracker.json.
  2. Prioritized Disk Read: When unchecking (deactivating) a package, the system attempts to read the list of tasks to be deleted directly from its local_meta.json on the disk and correctly removes them from the VS Code files.
  3. Emergency Brake (Deletion via File Explorer): If you previously deleted the package folder manually via File Explorer (meaning its local_meta.json no longer exists), VenvHub Pro automatically falls back to the data in the venvhub_tracker.json file, identifies the orphaned entries, and safely removes them from the VS Code configuration.
  4. Alignment During Venv Switching: If you switch the active environment within a single project (e.g., from Venv A with local packages to Venv B without packages), the background synchronizer automatically removes the tasks belonging to the old environment from the editor and prepares a clean interface for the new environment based on its configuration.
  5. Automatic Cleanup of Empty Configurations: If no active tasks or keybindings remain in VS Code after uninstalling a package or switching the Venv, VenvHub Pro physically deletes the tasks.json, keybindings.json, and the venvhub_tracker.json files, preventing the accumulation of empty configuration files in the project folder.

10.6 Bundling into PyInstaller (.EXE)

Due to the new no-UAC architecture (Import Hooks), local packages are not physically present in the environment folder. Standard PyInstaller would not automatically find them during static code analysis.

VenvHub Auto-Injector (Full Automation): When creating the final .exe program, you don't have to do anything extra regarding local packages in the PyInstaller Builder!

The VenvHub Pro Builder includes an intelligent pre-processor. Before launching PyInstaller, it automatically reads the venvhub.json file in your environment and generates in the background:
  • The --paths parameter, allowing PyInstaller to immediately find all your Python code from local packages (without needing to list "Hidden imports").
  • The --add-data parameter for every single linked package. The application automatically bundles all your images, .json configurations, and UI templates found within your modules directly into the resulting .exe.

10.7 Path Auto-repair on USB Drive (Portable Mode)

The new architecture makes transferring the application and projects on a USB drive extremely reliable and user-friendly.

Lightning-fast Silent Auto-repair: Whenever you launch the application on a new computer:
  1. VenvHub Pro detects that the drive letter has changed (e.g., from E:\ to F:\).
  2. It opens the venvhub.json file in your environment folder.
  3. Instantly and completely silently, it overwrites the old drive letters with the new ones.

Compared to the older version (which used Windows Junctions), a UAC window never pops up for repair, and the process completes in a fraction of a second.

10.8 Network Drives (NAS and Shared Folders)

✅ Full support for placement anywhere!

A huge advantage of the new "Hook" architecture (compared to outdated Hard Links) is that your "Local Packages Root" can be located truly anywhere.


Whether it's your local drive, a network drive (NAS), or a cloud folder – everything works without issues. Since VenvHub Pro writes paths only as plain text into a JSON file, the operating system (Windows/Linux) no longer cares about the type of drive where the data is physically stored.
