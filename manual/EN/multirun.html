5.1 Principle of Multi-run (Batch Launching)

Modern applications often consist of several parts that must run simultaneously (e.g., Backend + Frontend + Worker). Manually starting each one separately is tedious.

The Multi-run mode allows you to define "Groups". With one click of the Play button, the entire group starts at once, with each project running in its own, fully isolated process.

5.2 Creating and Populating a Group

Go to the Environment Manager window and navigate to the "Multi-run" tab.

  1. Create Group: In the left panel, click the + button and enter a name (e.g., "FullStack Dev").
  2. Select Members: In the middle panel, you can see a tree structure of all your projects and their environments. Check an environment (checkbox) to add it to the group.
  3. Save: Members will appear in the right table. Don't forget to click Save Changes in Group! Your settings will be written to the main configuration file.

5.3 Order and Scripts (Advanced Drag & Drop)

In the right table ("Group Members"), you can fine-tune the details:

Advanced Reordering (3-zone Drag & Drop)

The table order determines the launch sequence. If you need the database to start before the server, you can drag and drop the rows. The table uses an intelligent 3-zone system:

  • Insert: If you drop the mouse on the top or bottom edge of a target row, the dragged row is inserted before or after it.
  • Swap: If you drop the mouse in the center of a target row, the rows simply swap positions.

Changing the Launch Script

By default, the application uses the script you have set in the main "Environment Manager" tab for each project. However, if you need to run a different file for a specific project in this group (e.g., worker.py), you can simply double-click and overwrite the script name in the table.

Automatic Saving: After every reorder (Drag & Drop) or script edit, remember to click the Save Changes in Group button.

5.4 Launch and Error States

To start a group, switch the Mini-Bar to M (Multi) mode, select the group, and click Play ▶.

Conflict and Duplicate Run Prevention

The application includes protection that prevents a process from starting if it is already running. If you try to launch a group containing a project that is already active (e.g., started manually or belonging to another group), a warning will appear and the launch will be aborted to prevent conflicts.

Smart Restart

What should you do if one of the projects in a group crashes, but others keep running?

  1. Fix the error in your code.
  2. Press the Play ▶ button on the Mini-Bar again.

The application behaves intelligently:

  • It checks the entire group.
  • It detects that projects A and C are still running -> These projects are SKIPPED.
  • It detects that project B (which crashed) is not running -> This project is RESTARTED.

You don't need to shut down the entire group and start over. Just "re-launch" only the processes that crashed.

5.5 Process Ownership

In the background, important logic runs that assigns an "owner" to each process launch. When you start a group named "MyAPI", each process records in memory that it belongs to this group.

Why is this important?

Imagine you have two groups:

  • Group A: contains Project 1 and Project 2.
  • Group B: contains Project 2 and Project 3.

First, you start Group A. Projects 1 and 2 are running. Later, you decide to stop Group B. Without the ownership system, the application could accidentally shut down the running Project 2, even though you originally launched it as part of Group A.

Safety Lock: Thanks to the ownership system, VenvHub Pro ensures that when you press the Stop ■ button for a specific group, only and exclusively those processes launched under that group's header are terminated. Other running processes remain untouched.

5.6 Low-level Windows Stability (Job Objects and Handles)

VenvHub Pro includes a process management implementation tailored for Windows (utilizing native calls via the ctypes library). This architecture solves two critical problems of standard Python applications.

1. Prevention of "Hanging" Processes (Windows Job Objects)

Upon application startup, a system process container is created in the background – a Job Object with the JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE restriction set.

  • The registry assigns a system flag to each new subprocess and associates it with this Job.
  • If the parent application VenvHub Pro crashes unexpectedly, the Windows operating system internally detects the closure of the Job Object and immediately terminates all assigned child processes.
  • This ensures no orphaned processes remain running in the background, which would otherwise block ports or RAM.

2. Protection against Process Substitution (Windows Handles)

Windows recycles Process IDs (PID) relatively quickly. If a monitored process ends and the system assigns the same PID to another application, the registry could receive incorrect information.

  • VenvHub Pro opens and maintains an open system reference – a Process Handle – during process registration.
  • As long as this Handle is open, Windows guarantees that the given PID will not be recycled or assigned to another application until the registry safely closes it.
  • Process status detection is thus precise, eliminating the risk of the application accidentally monitoring or terminating another (foreign) program.
Safety First: All system references (Handles) are internally protected by Thread Locks and correctly closed during every stop, crash, or process cleanup, preventing memory leaks.
