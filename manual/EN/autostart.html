9.1 Hybrid Approach (Venv Orchestration)

The Container / Autostart tab is not a classic container system like Docker. It does not virtualize the entire operating system or network layers. Instead, it is a native Windows Orchestrator.

It utilizes lightweight Python virtual environments (Venvs) and combines them into managed units. The result is an extremely fast application startup with minimal memory consumption (no Docker Daemon running), while maintaining important features like auto-start, automatic restart after a crash, and RAM monitoring.

9.2 Adding a Group and In-time Saving

This tab is closely linked to the "Multi-run" tab. You cannot create new groups from scratch here; you must import them.

  1. First, create and assemble a project group in the Multi-run tab.
  2. Go to the Container tab and click on ➕ Add New Group.
  3. Select a group from the list. Its control panel (Widget) will be generated on the screen.
In-Time Saving (No Save Button): Unlike other parts of the program, everything is saved immediately in the Container. Any checkbox toggle or second-count overwrite is written to a dedicated .json file in a fraction of a second. The application is intelligent and adjusts the save path according to your mode: in the Portable version, it saves next to the program in assets/autostart_multi/, while in a regular installation, it saves securely to %APPDATA%/VenvHubPro/autostart_multi/.

9.3 Group Control Panel (LED, RAM, Modes)

Each imported panel serves as a separate control Dashboard for that group.

1. Visual Diagnostics

  • LED Diode: Indicates the status of the entire group in real-time.
    • Gray: No projects in the group (group is empty).
    • Red: The group contains projects, but all are currently stopped.
    • Green: At least one process is running and none are in an error state.
    • Flashing Orange: A crash detected in one or more processes. Flashes orange.
  • RAM Monitor: The application scans the operating memory every 2 seconds. It doesn't just count the master terminal but recursively adds up the memory of all subprocesses (python.exe and its threads), giving you an accurate figure of your group's hardware requirements.

2. Global Group Settings (Header)

  • Autostart: If checked, this group will automatically launch immediately after VenvHub Pro starts.
  • T (Terminal) vs S (Silent): Launch mode for all projects in the group.
    • T: A black window (CMD) opens for each process. Ideal for development and error tracking.
    • S: Processes run "invisibly" in the background. Ideal for production and bots.
Warning for Silent (S) mode: Although processes launch invisibly, VenvHub Pro utilizes the Windows Job Object mechanism, which automatically terminates all assigned subprocesses when the application is closed (even forced). Nevertheless, we recommend using the STOP button for a safe and clean group shutdown.

9.4 Timing, Anchor (Hook), and Chain Stability

VenvHub Pro v2.5.3 does not use a simple bulk start. Group launching is managed by a specialized thread (Launch Worker), which processes projects in strict top-to-bottom order and applies Chain Stability (each launched program monitors the stability of all its predecessors).

1. Intelligent "Wait Timer" (Delay before start)

Unlike standard scripts, the value in the Wait (s) column means a delay for the process start itself. For example, if you set 15 seconds for Program B:

  • The thread pauses, and a message ⏱️ Counting down Wait 15s... appears in the log.
  • The process (terminal window for Program B) opens only after this time has elapsed.
  • Chain Protection: During this countdown, VenvHub scans all previously running programs (e.g., Program A) every half-second. If Program A crashes during the countdown, the countdown for Program B is immediately paused and reset, and it waits until Program A successfully restarts!

2. Anchor (Hook) - Hard-link to application logic

The Anchor is the highest form of dependency management. If you activate it, VenvHub Pro stops being a "blind" launcher and starts communicating actively with the script:

  • Unique ID Marker: At each start, VenvHub generates a unique key and places its path into the VENVHUB_KOTVA environment variable.
  • Queue Blocking: The startup sequence for that project completely stops. No further program in the list will start until VenvHub sees that your script has physically created this file on the disk.
  • Post-crash Recovery (Anchor Recycling): If a program is restarted (Respawn), the same anchor path is reused (recycled), and the sequence waits for it faithfully again.
Full Time for Stability: If Program B was supposed to wait 15 seconds and was already at the 14th second, but Program A suddenly crashed, Program B's timer is reset. After Program A is fixed (and its new Anchor is received), Program B starts its 15-second countdown completely from zero to give the restarted Program A enough time to open ports and load data.

🛠 How to implement the Anchor signal in your code

To let VenvHub Pro know that your program is ready, you must insert a short block of code into your Python script. Upon startup, VenvHub passes the VENVHUB_KOTVA environment variable to your program, which contains the path to a unique signal file.

Insert the following code where your program finishes its initialization (e.g., after app.listen() or after loading data into RAM):

import os

def signal_ready():
    # 1. Check if VenvHub is requesting a signal
    hook_path = os.environ.get("VENVHUB_KOTVA")
    
    if hook_path:
        try:
            # 2. Confirm readiness by creating an empty file at the given path
            with open(hook_path, 'w') as f:
                pass
            print(f"⚓ Anchor: Signal sent successfully.")
        except Exception as e:
            print(f"⚠️ Anchor: Error sending signal: {e}")

# Call this function when your program is 100% up and running
signal_ready()
What happens after sending? VenvHub Pro immediately detects the creation of this file, deletes it (to clean the system), and instantly starts the countdown (Wait) for the next project in line.

9.5 Collapse Detection and Domino Effect

The system contains two levels of protection against application crashes, which behave differently depending on the project's phase.

1. Domino Effect on Predecessor Failure

The "One for All" architecture means that the youngest (currently starting) program constantly monitors all older programs above it. If any predecessor crashes during startup or waiting:

  • Watchdog Monitoring: Even during blocking wait (Wait) or Anchor phases, the Watchdog constantly checks the PIDs of all processes already started in the sequence.
  • Grace Period (6s): During the Anchor phase, the system ignores the status for the first 6 seconds to give slow terminal startups and python.exe creation a chance.
  • Fatal Break (Critical Failure): If a predecessor crashes 3 times in a row and exhausts its restart limit, the sequence is permanently blocked! VenvHub outputs 🛑 FATAL ERROR and will not start any further projects in order. This prevents errors where Program B would fail because Program A cannot start.

2. Runtime Phase (Respawn Logic and 3-Strike Rule)

If a project has successfully passed the Anchor phase and a crash occurs later during normal operation, the Respawn Manager is activated:

  • Incident Logging: Each crash is recorded in crash_log.jsonl directly in the venv folder (contains timestamp and status).
  • Hard Cleanup: The system waits 2 seconds, forcibly kills the dead terminal window (if it remained hanging), and releases ports.
  • Reincarnation: The project is restarted with the same cwd (current working directory) to avoid path errors with exclamation marks and spaces.
  • 3-Strike Rule: If a project crashes 3 times in a row, the system declares it unstable, blocks further Respawn attempts, and leaves the window (if there was one) available for developer error analysis.
Important Notice: If the STOP button on the group panel is manually clicked, all crash counters are zeroed and the "3-Strike Rule" is reset for all programs.

9.6 Application Boot and Ownership Registry

Process on VenvHub Pro Startup

When you open the main application, a special service called AutostartBooter scans all your Containers. If it finds a group with Autostart checked, it attempts to launch it. Before doing so, it performs a conflict check:

  • If it detects that some processes from this group are already running in Windows (e.g., because you closed VenvHub without stopping the processes and are now opening it again), Autostart is blocked just in case to prevent double-start attempts and port collisions.

Process Ownership Registry

You can have the same project included in two different groups (e.g., in a "Testing" group and a "Production" group). If you launch the Testing group, the application records in internal memory (the "Registry") that this running process is owned by the Testing group.

If you subsequently try to click "Stop" in the Production group, the system detects that this group is not the owner of that process and refuses to terminate it. Through this mechanism, the application guarantees that operations of one group never interfere with the operation of another group.
